# 操作手册模块骨架规范

操作手册中每个功能模块按以下标准结构编写。强制要求：**截图前置、四表覆盖、不含"等"字**。

## 一、模块骨架

每个模块按以下结构组织，不可省略任何一级：

```
### X.X 模块名称

操作路径：<从项目菜单数据中提取的真实路径>。

**操作路径写入规则（强制）**：
- 必须从项目的菜单数据（SQL `sys_menu` INSERT 语句或前端路由 `meta.title`）中提取真实层级。
- 禁止根据页面源码目录结构推测或编造菜单路径。
- 禁止在操作路径中附带权限标识（如 `system:contractorcompany:list`）或技术细节——这些信息面向开发人员，软著审核员不需要也不理解。
- 操作路径格式：`一级菜单 → 二级菜单`，仅包含菜单名称，用中文箭头分隔，不写"左侧菜单""点击""进入"等动作描述。
- 无法从项目数据中提取时，使用"从系统菜单进入对应功能页面"作为兜底表述。

模块用途简述。（1-2 句，说明该页面解决什么业务问题）

[该模块的 PlantUML 操作流程图或截图——提前到文字之前]

[截图：模块列表页 — 放在操作流程文字之前]

#### 一、列表界面

| 功能点 | 描述 |
|---|---|
| 列表展示字段定义 | 逐列列出所有表格列，每个列注明含义。**不准用"等"字省略任何列** |
| 数据排序 | 声明默认排序规则（如"默认按创建时间降序排列"），不可省略 |
| 默认每页数据量 | 声明每页展示条数（如"10 条/页"），不可省略 |
| 查询功能 | 逐一列出所有筛选条件及其匹配方式（模糊/精确） |
| 新增 | 按钮名称和触发动作 |
| 修改 | 触发途径（行操作按钮或详情页入口） |
| 删除 | 单条删除和批量删除的能力声明 |
| 批量操作 | 除删除外的其他批量操作（导入/导出/状态切换等），如无则写"无" |

#### 二、新增/修改界面

| 字段名称 | 字段类型 | 是否必填 | 默认值 | 输入边界 | 业务规则 |
|---|---|---|---|---|---|
| 字段1 | 文本输入/下拉选择/日期选择器/复选框/单选按钮 | 是/否 | 如有默认值 | 长度限制/数值范围/格式要求 | 唯一性校验/关联约束/触发条件 |
| ... | ... | ... | ... | ... | ... |

**字段类型**必须精确：文本输入、下拉选择、日期选择器、时间选择器、数值输入、复选框、单选按钮、人员选择、部门选择、树形选择。
**是否必填**必须标注条件触发——如"否（勾选'是否允许提前巡检'时必填）"。
**输入边界**必须给出具体数值或格式——如"1-720（分钟）"、"不超过 32 字符"、"格式为 yyyy-mm-dd"。
**业务规则**必须写清校验逻辑和触发条件——如"唯一性校验，重复提示'名称重复，请调整后保存'"。

#### 三、异常功能逻辑

| 异常情况 | 处理逻辑 |
|---|---|
| 具体的异常场景（精确到字段级别的重复、格式错误、关联约束等） | 系统响应（提示文案、阻止/允许、后续操作指引） |

至少覆盖 4 种异常：**数据重复**、**关联约束阻止删除**、**必填字段为空**、**网络中断**。如有模块特有异常（如时间逻辑校验、文件格式校验）也需列出。
```

## 二、代码组织

四个子节标题固定为中文数字序号（一～四），与正文的一二三四层级隔离。
如某模块有多个独立界面（如"巡检计划"有基本信息/周期规则/推送与路线三个子界面），可拆为多个"新增/修改界面"子节，编号为一～五：

```
一、列表界面
二、新增/修改界面（基本信息）
三、新增/修改界面（周期规则）
四、新增/修改界面（推送设置与巡检路线）
五、异常功能逻辑
```

## 三、App 端模块的差异

App 端功能操作不使用"四表一图"骨架，改为"操作步骤表"：

```
### X.X 模块名称

功能用途简述。（1-2 句）

[该模块的 PlantUML 操作流程图]

#### 操作步骤

| 步骤 | 用户操作 | 系统响应 | 异常处理 |
|---|---|---|---|
| 1 | 具体操作 | 界面反馈 | 异常场景和处理 |
| 2 | ... | ... | ... |
```

App 端模块通常涉及多个界面切换和硬件交互（NFC/摄像头/GPS），用操作步骤表比字段表更合适。

## 四、功能清单表要求

操作手册的功能清单表（文档层面，汇总所有模块）必须满足以下密度要求：

| 模块名称 | 功能概要 | 详细说明章节 |
|---|---|---|
| 模块1 | 逐条列出该模块的子功能，用分号分隔。**每行至少列出 5 项子功能**，不可用一句笼统概述替代。每条子功能必须是可独立操作的功能点（如"上传营业执照""发起审批""管理黑名单状态"），不可写为抽象目标（如"实现承包商管理"）| X.X |
| ... | ... | ... |

**示例（正确）**：
"登记公司基本信息（名称、编号、类型、信用代码、负责人等）；上传营业执照、合同、安全协议和资质证书；发起备案审批，跟踪审批状态；查看公司详情和备案状态；管理黑名单状态和原因；支持查询、新增、修改、删除、批量导入导出"

**示例（错误）**：
"登记承包商公司基本信息，上传营业执照、合同、安全协议和资质证书；支持查询、新增、修改、删除、详情查看和批量导入导出" — 仅 2 项子功能，远低于 5 项要求。

## 五、FAQ 最低要求

FAQ 章节至少覆盖以下条目，不可低于 8 条：

**Web 管理端（3+）**：
- 删除操作被关联约束阻止 → 如何解除关联后重试
- 条件字段的触发逻辑 → 什么情况下该字段出现/消失
- 批量操作的范围 → 勾选记录的作用范围

**App 巡检端（3+）**：
- 签到失败 → 切换签到方式/联系管理员
- 网络中断数据丢失 → 暂存机制
- 强制按序中途受阻 → 管理员应急调整

**跨岗位（2+）**：
- 不同角色看到的数据范围不同 → 权限隔离说明
- 随手拍/异常上报后如何追踪处理结果 → 跨端闭环路径

## 六、禁止事项

- **列表字段不准用"等"字**——要么列全，要么不列
- **字段表不准合并技术细节**——字段类型、输入边界、业务规则必须分列
- **异常逻辑不准散落在段落中**——必须是独立的异常功能逻辑表
- **功能清单不准每行低于 5 项子功能**——功能概要必须逐条列出可独立操作的功能点，不准一句话模糊概括
- **FAQ 不准少于 8 条**
- **截图不准放在章节末尾**——必须先截图再文字（证据前置）
- **不准省略数据排序和默认每页数据量**
