# 业务理解规则

申请表信息和操作手册不能只根据代码结构泛泛生成，必须先理解软件业务。

## 证据收集

先用脚本收集证据，输出 `草稿/业务理解证据.md/json` 和 `草稿/业务理解模型稿模板.json`。证据通常包括：

- `README.md`
- `docs/*PRD*.md`
- `docs/*BRD*.md`
- `docs/*ARCHITECTURE*.md`
- 产品说明、需求文档、设计文档
- 前端页面标题、按钮文案、路由、核心组件名
- 后端 API 路由和模型名称

这些只是候选证据，不代表最终行业、功能和手册结构。

## 输出业务理解草稿

模型必须阅读证据和必要源码，自行判断应该抽取哪些业务信息，再生成业务理解模型稿 JSON。不得用关键字表或固定模板决定行业、功能和结构。

模型稿经脚本校验后生成 `草稿/业务理解.md` 和 `草稿/业务理解.json`，至少包含：

- 产品组成与闭环验证：使用 `product_composition` 声明各端、目标用户、代码仓库和模块目录；使用 `closed_loop_validation.chain/node_mapping/conclusion` 验证业务链每个节点均有对应模块、角色和结果。
- 产品定位
- 面向领域 / 行业
- 目标用户
- 用户痛点和核心价值
- 主要业务功能
- 典型操作流程
- 操作手册结构建议
- 操作手册页面/流程模块，必须说明每个真实页面或核心流程的使用场景、进入位置、用户可见元素、用户动作、输入/状态规则、结果反馈和截图预留
- 申请表建议口径
  - `application_purpose`：开发目的，不少于 100 字。
  - `main_functions`：软件的主要功能，**2026 年新政要求 500-1300 字**，必须覆盖研发背景、核心技术架构、功能模块及应用场景。
  - `technical_characteristics`：技术特点描述，不少于 100 字。
  - 以上三个字段合计应满足申请表"主要功能描述"500-1300 字的最低要求。
- 证据来源
- 待用户确认项

## 外部调研

如果项目材料不足、业务类型较新，或用户明确希望参考竞品，可联网搜索相近产品和行业资料。

外部调研只用于帮助理解行业表达，不能编造项目不存在的功能。需要把调研结论写入业务理解草稿，并区分“项目证据”和“行业参考”。

## 模块分类与填写

在填写 `manual_modules` 之前，模型必须先阅读每个模块对应的源码，判断模块操作类型——台账型、业务型或混合型。分类规则和对应的 JSON 结构见 `references/module_classification_rules.md`。关键原则：

- **台账型**（纯CRUD，如设备管理、NFC、二维码）→ 用 `registry` 结构，逐项枚举 columns/filters/actions/form_fields。
- **业务型**（有状态流转或操作链路，如巡检计划、任务执行、签到）→ 用 `business_operation` 结构，描述对象生命周期和操作链路。
- **混合型**（台账面+业务配置面，如巡检点管理）→ 同时填 `registry` 和 `business_operation`，每条配置路径分别列出。

### 完整枚举强制要求

台账型模块的 `columns`、`filters`、`top_actions`、`row_actions` 必须逐项枚举，不准用"等"字省略。这是之前实际操作中反复出现的问题——模型倾向于把 30+ 字段的表单压缩成 5 个字段的摘要。完整枚举是操作手册质量的底线保证。

### 业务型模块的操作链路强制要求

业务型模块的 `operation_chain` 必须按用户操作的时间顺序逐阶段展开。每一个条件分支（如"选择固定周期→激活 A 表单，选择自定义周期→激活 B 表单"）必须在 `conditional_branches` 中说明两条路径的差异。不准把多路径操作用单线叙述糊过去。

### 配置链路验证

混合型和业务型模块必须标注 `entry_conditions`，并在配置步骤中标注 `references`——说明该步骤引用了哪个台账模块的数据。这样在审查业务理解时，可以通过 references 快速验证配置链路是否正确（例如：巡检点引用NFC→计划引用巡检点→任务由计划生成）。

## 业务主线与逆向场景

业务理解文档的开头必须包含一个自然段描述业务主线流，然后按系统作用域分层展开逆向调整与异常处理场景。这两段共同构成业务理解的叙事核心，让读者在进入模块清单之前先建立"这个系统在管什么、出问题时怎么办"的整体认知。

### 业务主线段

用一段自然文字描述系统从起点到终点的完整正向流程。必须串联核心业务对象在各阶段的流转关系，点名各角色在什么阶段做什么事、产生什么结果。不要写成功能列表，要写成"谁→做什么→产出什么→谁接着用"的连贯叙述。

### 逆向场景段

任何业务系统都管理若干核心对象在其生命周期中的状态推进。正向是状态递进，逆向是状态回退、修正、旁路或终止。逆向段按系统作用域分层展开，每层一段。这六层是 MECE 的通用框架，适用于任意业务领域：

1. **配置与基础数据层** — 分类标签、模板、规则参数、参考数据被修改或删除后，已引用这些基础数据的下游对象如何同步、重新关联或兼容。
2. **核心业务对象层** — 系统管理的主要实体（培训任务、工单、申请、计划等）在发布后如需修改内容、更换关联子对象或调整规则，如何通过下架→编辑→重新发布或终止等操作完成；进行中的对象能否暂停或终止，已产生的中间记录如何处理。
3. **参与者与范围层** — 业务对象的参与主体（人员、组织、角色）在周期内发生入职、离职、调动、退出时，已分配的可见范围和待办任务如何增减或批量移除，调整后前端可见性如何同步。
4. **执行过程层** — 用户在消费或执行过程中出现中断、超时、未完成时，系统是否保留进度、是否允许断点续做、是否允许延期；超过截止时间后是自动标记还是允许补做。
5. **结果与判定层** — 结果产出后如需纠偏：补做、重新判定、争议复核、批量修正导入、作弊或违规作废。修正后原记录是保留作为追溯依据还是覆盖。
6. **归档与终态层** — 业务对象进入终态后：重新激活还是永久锁定；证书或证明文件的生成、修正、作废与补发；归档数据的查询、导出和保留期限。

模型在编写业务理解时必须：

- 用真实项目证据填充这六层中与当前项目相关的部分。
- 每层只写跟当前项目实际存在逆向路径的内容；如果某层完全是单向不可逆的，写一句"本层为单向不可逆，无逆向路径"即可，不要编造。
- 不要用各层的标题原文当段落开头；每段的第一句应自然说明当前层管理什么对象、在什么场景下需要逆向操作。
- 逆向段的写作目标是"让读者理解这个系统不是只走阳光大道，而是有完整的异常处理和变更管理能力"，不是罗列功能清单。

- `申请表信息.md` 的开发目的、行业、主要功能、技术特点必须优先来自业务理解。
- `操作手册.md/docx` 的说明、功能特点、系统要求、核心页面/流程、常见问题、术语表和章节结构必须优先来自模型确认后的业务理解；典型流程必须由 `operation_flow[].step/result` 提供动作和结果。
- 操作手册不应只生成抽象”功能列表”。模型应把路由、页面、按钮、输入框、列表、弹窗、状态提示、额度或权限规则等用户可见证据整理到 `manual_modules`，供脚本按通用操作手册骨架排版。最终成稿应是段落化用户手册，不是”进入方式/页面内容/操作步骤/结果反馈”的字段列表。
- 如果缺少 `manual_modules`、`system_requirements`、`faq` 或 `glossary`，应回到业务理解阶段补充真实内容；脚本不得用分类模板兜底生成。
- 如果业务理解仍不充分，先提示用户补充产品说明，而不是直接生成泛泛描述。

## 输出质量的核心驱动

操作手册的最终质量取决于业务理解 JSON 的内容丰富度。操作手册由模型直接撰写，验证脚本只检查内容质量和证据缺口，不生成或补写正文。

文本来源分两路：

- **主路径**：`operation_flow[].step/result`、`manual_modules`、`registry`、`business_operation` 或 `crud_scenarios` 等结构化字段，由模型在阅读项目页面后写入，并作为操作手册正文的事实依据。
- **证据缺口路径**：仅记录缺失字段和补全线索。若验证发现证据缺口，`操作手册审查报告.json` 会在 `self_review.rounds[].evidence_gaps` 中记录，模型必须回到业务理解 JSON 补齐对应字段。

禁止链路：

```text
关键词识别 -> 套用硬编码句子 -> 写入操作手册正文
```

允许链路：

```text
关键词识别 -> 定位页面/代码/字段/接口/状态证据 -> 补全业务理解 JSON -> renderer 渲染结构化字段
```

### crud_scenarios 字段结构

每个 `manual_modules` 中的模块应包含 `crud_scenarios`，按场景分组，每个场景下为步骤数组：

```json
“crud_scenarios”: {
  “查询与查看”: {
    “steps”: [
      {
        “action”: “用户在<具体控件>中输入<条件>后点击查询按钮”,
        “system_response”: “系统<页面刷新/列表变化/状态更新>的具体描述，包含可见的数据变化”,
        “error_handling”: “<具体异常场景>时，用户应<具体操作>；若仍失败则<下一步>”
      }
    ]
  },
  “新增与编辑”: { “steps”: [...] },
  “删除”: { “steps”: [...] }
}
```

列含义：
- `action` — 用户的自然操作描述，含控件名、输入内容、点击目标，不脱水；不少于50字符
- `system_response` — 系统反馈，描述页面可见变化，具体到列表刷新了哪些数据、弹窗显示了什么；不少于40字符
- `error_handling` — 异常处理，按真实可能的异常场景写，不要”联系管理员”万能句；不少于30字符

各组不必全部出现，只写当前模块真实存在的场景即可。

### 步骤拆分原则

一个业务场景拆分为多少步的判定依据是**该场景中用户操作的物理断点**——每次用户需要”停下来判断、停下来等系统响应、停下来处理不同异常”的地方就是一个步界。

常见拆分模式：

- 准备/下载阶段 → 独立 step（如：下载模板、打开工具）
- 数据填写/整理阶段 → 独立 step（含字段规则和填写要求）
- 提交/校验阶段 → 独立 step（含校验逻辑和校验结果格式）
- 结果处理/异常修正阶段 → 独立 step（含修正流程和约束条件）

反例——以下写法是”单步吞没全场景”（禁止）：

```
“action”: “用户完成成绩录入后提交”
```

应拆分为：

```
“action”: “用户逐人在分数框中填写成绩...” → “用户点击下载模板批量填写...” → “用户核对汇总后提交” → “提交后发现错误联系管理员覆盖”
```

对于任意新业务领域，只要遵循拆分原则，生成器就能在操作表中呈现足够密度的步骤信息。
