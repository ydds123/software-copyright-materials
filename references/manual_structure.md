# 操作手册结构

操作手册应像真实软件随附的操作说明，目标是让读者知道软件用途、功能和基本操作。

推荐采用软著审核友好的通用骨架，类似传统操作手册：

1. 系统简介：说明软件定位、目标用户、业务场景和整体流程，按功能维度逐条展开。
2. 功能特点：按当前项目真实功能概括 4-8 项特点，每项说明业务作用和用户可见结果。
3. 系统要求：用表格说明最低配置和推荐配置；Web 项目可写浏览器、网络和服务访问要求，桌面项目可写操作系统、处理器、内存、存储和分辨率，移动端项目写操作系统版本、网络、存储空间。
4. 系统架构概述（2026 新政建议）：如项目有完整的分层架构，可用一段文字和一张架构层级图描述系统的整体技术架构（如客户端→接入层→业务层→数据层）。不要求画 UML，但需说明各层职责、核心模块和层间关系。此章节目的是让审核员建立对系统整体技术方案的认知，同时满足 2026 年"设计说明书需包含软件结构图、功能流程图"的补充审查要求。
5. 术语表：解释软件名称、核心业务对象、页面模块和用户可能不熟悉的术语。前置到此处，让读者在进入具体功能操作章节前建立共同词汇基础。
6. 具体页面 / 功能操作：从前置术语表章节之后开始，按真实页面、导航入口或核心流程逐章说明，每章写使用场景、页面用途、进入位置、页面内容、用户动作、输入限制或异常提示、操作结果和截图预留。
7. 典型使用流程：如项目存在清晰串联流程，可单独写一章串起从进入软件到完成核心任务的过程。
8. 常见问题解答：写 3-5 个与当前软件真实使用相关的问题和解决方法。

以上是通用骨架，不是旧项目内容。正式章节标题使用中文大写序号，例如 `一、系统简介`，不要使用 `(1)、系统简介`。生成时必须根据当前项目业务、页面入口、功能关系和用户可见控件填充内容，不要照抄用户提供的范本文案。
具体页面和流程必须来自 `草稿/业务理解.json` 中模型写入的 `manual_modules`。如果该字段为空，不能根据 `business_features` 生成兜底模块，应停止并要求模型阅读真实页面和项目资料后补全。典型流程必须来自 `operation_flow` 的结构化对象列表，每项包含 `step` 和 `result`；不要让渲染器根据动作文字临时推断结果。

## 写作口径

- 使用通用、客观、简洁的中文。
- 不写面向终端用户的复杂教程。
- 每个模块遵循”功能介绍→操作路径→功能说明→操作步骤表”结构。不写页面元素说明表。
- 核心是操作步骤表，四列：操作步骤（序号）、用户操作（用户实际动作）、系统响应（系统反馈什么）、异常处理（出错时用户怎么办）。
- 不同模块功能说明的篇幅根据复杂度自然变化，简单的 2-3 句即可。
- 每个章节必须有段落化说明，不能只写项目符号列表。
- 每个功能模块必须覆盖”页面用途 + 用户动作 + 系统反馈 + 异常处理”，通过操作步骤表清晰呈现。
- 补充说明段落只能写当前软件的用途、业务场景、页面组织和用户流程，不写“本操作手册用于……”“面向软著审核……”“不描述代码实现……”这类解释文档写作方式的元话语。
- 禁止在脚本中按 auth、query、form、workflow 等分类自动生成入口、步骤或结果反馈。入口和步骤必须来自模型对当前项目真实页面的阅读。
- 功能特点不要写成“开头一句总述 + 编号列表 + 结论”的头中尾结构。每个特点用段落展开，说明该功能解决什么业务问题、用户在页面上看到什么、完成操作后得到什么结果。不同功能的说明要有差异，避免每条都使用相同句式。
- 页面章节不要输出“进入方式：”“页面内容：”“操作步骤：”“操作规则：”“操作结果与反馈：”这类模板小标题。
- 操作手册语言要让审核员和普通读者能看懂，重点说明模块是做什么的、怎么操作、操作后看到什么。避免代码、框架、接口、状态管理、异步任务等技术化表达。
- “AI 味”主要表现为空泛、整齐、万能、没有项目现场感：例如每个模块都用同一种句式，反复写“提升效率、优化体验、提供支持”，使用“旨在、赋能、一站式、智能化、高效便捷、显著提升、强大能力、丰富功能”等口号，却没有说明当前项目的真实页面、真实对象、真实动作和真实反馈。发现这类内容时必须改写成朴素、具体、可回溯到项目证据的表达。
- 截图前必须先让用户在 Chrome DevTools MCP、Codex Computer Use、用户自行截图三种方式中选择；选择后检查对应能力是否可用。**2026 年新政下，操作手册截图已从"建议"升级为强制要求**——必须包含完整的登录界面及操作步骤截图，截图中的软件名称、公司 Logo 必须与申请表信息保持绝对一致。不再提供"跳过截图"作为推荐的默认选项；若用户确实无法截图（如项目未运行），可保留 `【截图预留：请在此处插入 xxx 页面或操作结果截图。】` 占位符，但需在生成报告中明确标注截图缺失可能导致的补正风险。

- 不夸大不存在的功能。
- 功能名称和章节组织由模型根据项目证据判断；路由、页面、README、接口和组件命名只是证据来源，不是固定抽取规则。
- Markdown 草稿生成前由 agent 自行检查章节完整性、内容厚度、项目流程一致性和语言自然度，发现章节过薄、模块套话、AI 味、技术化表达或相邻模块含义混淆时先循环补写和修正；完整草稿生成后只向用户发起一次整体确认，再进入 Word 生成。
- 操作手册生成时必须同步输出 `操作手册自检记录.md/json`。记录中至少包含初稿生成、按项目流程扩写、去除制式表达和 AI 味三轮；如果第三轮仍发现问题，要继续自动修正并追加轮次记录。自检 JSON 中的 `fallback_usage.count` 用于审查是否仍由 fallback 生成正文内容，正常情况下应为 0。
- 操作手册必须基于已确认的业务理解写作。相近功能应结合项目真实业务分别说明各自的操作目的、用户动作和结果反馈，不能用同一段话替换不同模块。
- 自检时必须检查是否生成了“功能操作说明”大章下反复套同一批模块的情况；如果出现同一模块重复多次，必须改为每个真实页面独立成章。
- 禁止把测试项目中的行业、角色、流程、功能名称或示例文案写成通用规则；范本只能帮助理解软著手册需要“通顺、具体、能给审核员看懂”，不能作为固定内容来源。
- 适用用户表应体现不同角色的职责差异，不能给所有用户写同一句“查询、维护、提交或查看相关业务数据”。详细门禁见 `manual_quality_gates.md`。
- 表格中的“用途说明”“处理方式”“预期结果”“结果反馈”等列必须有信息增量；同一表格同一列重复出现 2 次及以上时，应按当前行字段、动作、规则或业务阶段重写。详细 5Y 分析和处理策略见 `manual_quality_gates.md`。
