# 操作手册质量规范

本文件是操作手册内容质量审查规则的唯一真相来源，只回答“怎么审”。正文构建规则见 `manual_authoring_spec.md`。

## 1. 通过标准

操作手册必须同时通过四个互斥维度：

1. 真实性：内容可以回溯到当前项目证据。
2. 完整性：读者、章节、模块、流程和截图覆盖充分。
3. 一致性：术语、状态、功能声明和操作说明互不冲突。
4. 表达质量：内容具体、可读、有信息增量且无明显 AI 或技术泄漏。

自动检查返回 0 errors 后，仍必须完成模型语义审查；自动检查不能代替真实性判断。

## 2. 真实性审查

### Q-T01 功能真实性

- 检查对象：功能清单、功能操作章节、FAQ。
- 通过标准：所有功能、按钮、字段、状态和反馈均有 `业务理解.json` 或 `manual_modules[].evidence` 支撑。
- 检查方式：模型语义审查。
- 失败处理：回读证据源码，修正文档或回到业务理解阶段补全证据。

### Q-T02 操作路径真实性

- 检查对象：每个模块的操作路径。
- 通过标准：来自菜单 SQL、路由标题或明确兜底表述，无英文路由名和编造路径。
- 检查方式：模型语义审查。
- 失败处理：修正路径；无法确认时使用统一兜底表述。

### Q-T03 截图真实性

- 检查对象：截图、截图清单和截图引用。
- 通过标准：截图属于当前软件、当前模块，名称和归属信息一致，文件存在且清晰。
- 检查方式：模型视觉审查与自动存在性检查。
- 失败处理：替换、移动截图或保留明确预留。

## 3. 完整性审查

### Q-C01 全文章节完整

- 通过标准：系统简介、系统概述、功能清单、系统要求、术语表、功能操作、典型流程和 FAQ 均存在且有正文。
- 检查方式：自动检查与模型审查。

### Q-C02 模块覆盖完整

- 通过标准：`manual_modules` 中每个申报模块都有操作章节；业务型模块具备状态或生命周期、条件分支和操作步骤。
- 检查方式：自动检查与模型审查。

### Q-C03 读者覆盖完整

- 通过标准：目标用户中的每个角色均有明确场景、操作路径和结果说明；读者覆盖矩阵中的问题均有正文回答。
- 检查方式：模型语义审查。

### Q-C04 截图覆盖完整

- 通过标准：登录界面和主要业务页面有截图或清晰预留；不把弹窗和同页状态过度拆分为独立截图。
- 检查方式：自动数量检查与模型审查。

## 4. 一致性审查

### Q-I01 术语一致

- 通过标准：正文不出现术语标准表中禁止的别名。
- 检查方式：自动扫描。
- 对应实现：`content_quality_check.py` 术语一致性扫描。

### Q-I02 功能清单与详情一致

- 通过标准：功能清单中的每个子功能在详情中存在；详情中的核心能力没有遗漏在功能清单之外。
- 检查方式：交叉引用报告与模型语义审查。

### Q-I03 状态机闭环

- 通过标准：业务型和混合型模块的每个状态有明确入口、出口、操作者和异常路径。
- 检查方式：模型语义审查。

### Q-I04 FAQ 与正文一致

- 通过标准：FAQ 中的处理方法、按钮、状态和异常说明与正文一致。
- 检查方式：模型语义审查。

### Q-I05 章节职责互斥

- 通过标准：同一信息只在一个章节展开，其他位置使用交叉引用，不跨章重复大段内容。
- 检查方式：模型语义审查。

## 5. 表达质量审查

### Q-W01 面向用户

- 通过标准：核心模块体现角色、场景、用户动作和结果反馈，不以申报摘要或研发说明口吻写作。
- 检查方式：自动角色关键词检查与模型审查。

### Q-W02 表格信息增量

- 通过标准：同一表格同一列不存在重复的泛化内容；每行帮助读者作出不同判断。
- 检查方式：自动重复检测与模型审查。

### Q-W03 无 AI 套话和模板化表达

- 通过标准：无空泛赞美、营销口号、万能结尾和跨模块重复句式。
- 检查方式：自动黑名单与模型审查。

### Q-W04 无技术泄漏

- 通过标准：正文不含源码标识符、API 路径、框架名、SQL、内部数据字典键名或未翻译技术实现。
- 检查方式：自动正则检查与模型审查。

### Q-W05 图片位置正确

- 通过标准：图片和截图与所说明内容紧邻，不堆在模块末尾，不跨章节归属。
- 检查方式：自动位置检查与模型审查。

## 6. 审查产物

主审查产物为 `草稿/操作手册审查报告.json`，其中必须包含：

- `self_review`：自动和模型自检轮次。
- `cross_reference.checks`：功能清单、业务声明、术语、状态、截图、按钮字段和 FAQ 七项交叉验证。
- `semantic_consistency.checks`：前后表述、状态机、角色路径、FAQ 矛盾和功能清单详情五个维度。
- `reference_comparison`：用户指定参照手册时记录对比结论。

迁移期间允许读取旧版 `交叉引用验证报告.md`、`语义一致性审查报告.md`、`操作手册自检记录.md` 和 `操作手册自检记录.json`；统一审查报告存在时，以统一报告为准。所有检查项必须为通过状态，不得保留未解决问题。

## 7. 自动检查

运行：

```bash
python3 ${CLAUDE_SKILL_DIR}/scripts/content_quality_check.py --manual 草稿/操作手册.md
```

自动检查覆盖技术术语、章节编号、表格密度、功能清单、截图覆盖、AI 套话、角色贯穿、多端覆盖、功能对等、表格信息增量、术语一致性、截图位置、代码泄漏和内部标识符泄漏等形式规则。

自动检查未实现或不能可靠实现的内容必须由模型审查：

- 源码证据是否真的支持正文。
- 章节职责是否语义重叠。
- 读者覆盖问题是否真正得到回答。
- 截图内容是否与文字匹配。
- 状态机和跨角色流程是否逻辑闭环。

## 8. 条件参照审查

用户指定参照手册时，运行：

```bash
python3 ${CLAUDE_SKILL_DIR}/scripts/extract_reference_profile.py \
  --reference "<参照手册路径>.md" \
  --out analysis/reference_profile.json

python3 ${CLAUDE_SKILL_DIR}/scripts/compare_reference.py \
  --reference "<参照手册路径>.md" \
  --target "草稿/操作手册.md" \
  --out "草稿/参照手册对比报告.json" --strict
```

参照手册只用于比较结构、密度和表达深度，不得成为当前项目功能真实性依据。
