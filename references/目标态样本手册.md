# 目标态样本手册

本文件从已通过软著审核的多份操作手册中抽取最有代表性的章节作为正样本，供模型撰写新操作手册时对标。来源覆盖了软件著作权申请中的典型系统形态：

| 来源系统 | 端形态 | 代表特征 |
|----------|--------|----------|
| 供应商准入管理系统 | 纯 Web | 审批流转、状态机、十模块快照、多角色签批 |
| 区域人员定位管理系统 V1.0 | Web + 大屏 | IoT 硬件、实时地图、大屏态势、出入闭环 |
| 设备设施巡检管理系统 V1.0 | Web + Android App | 四级检查体系、计划排程、隐患治理闭环 |
| 作业安全监控管理系统 V1.0 | Web + Android App | XYZ 优先级矩阵、报警规则配置、看板钻取、App 处置 |

**使用方式**：模型在编写操作手册前，先阅读本文件中与目标模块类型对应的样本章节，理解叙事密度、表格深度、状态流转表达方式和角色贯穿方式，再动笔。

## 节选一：登录界面

> 登录界面放在 Web 管理端功能操作章内作为开篇子节。它是用户在 Web 端的实际入口体验，不应作为独立章与系统简介/概述平级。如果系统存在独立首页（含工作台、统计区、待办区等功能布局），登录界面之后可追加系统首页子节。

**样本原文**：

### 6.1 登录界面

【截图预留：登录界面截图】

界面介绍：登录页是系统的入口。用户在浏览器地址栏输入系统地址后进入登录页面——输入账号、密码和验证码后点击登录进入系统。

- 登录账号：由系统管理员在「系统管理」→「账号管理」中对账号进行创建和管理。用户如需新增或变更账号权限——请联系系统管理员。
- 账号密码：初始密码由系统管理员分配——用户首次登录后建议立即修改密码。若忘记密码可联系系统管理员重置密码。
- 验证码：登录页面展示图形验证码——输入错误时点击验证码图片可刷新。

登录成功后系统进入首页——首页左侧菜单栏按业务分类展示各功能模块入口（用户可见范围由系统管理员在角色权限中配置），右侧展示当前用户的待办任务和关键数据统计。若登录失败——页面显示账号不存在、密码错误或账号已停用等提示，用户应根据提示修正后重新登录。

**对标要点**：
- 一张截图占位 + 账号/密码/验证码三要素逐条说明
- "界面介绍"一句话总述
- 登录失败三种提示（账号不存在/密码错误/账号停用）全部覆盖
- 登录成功后连接到首页——形成"入口→校验→进入→失败处理"的完整闭环
- 不写技术细节（不写 OAuth/SSO/JWT/Session/Token）

## 节选二：系统简介（第一章）

> 第 1 章的目标是让一个不了解该系统的人读完后知道：这系统干什么、谁在用、核心业务链路是什么。

**样本原文**：

作业安全监控管理系统是面向化工企业特殊作业场景的安全报警管理平台，系统围绕动火、受限空间、高处、吊装等特殊作业票的执行过程，对作业人员离场、无关人员闯入、气体检测超时和作业票超期四类安全风险进行实时监测和分级报警。系统由 Web 管理端和安卓 App 端组成，面向安全管理员、控制室值班人员、现场管理人员和企业负责人四类用户，其主要功能如下：

报警优先级基础标准建立。安全管理员在报警优先级配置页面定义优先级矩阵——通过维度配置对话框分别设置 X 轴（报警类别维度）和 Y 轴（响应时效维度）的取值列表，再通过报警级别配置对话框设置 Z 轴（优先级取值）的颜色编码级别，最后在矩阵表格中为每个"Y 取值 × X 取值"交叉点选择对应的 Z 轴优先级值。矩阵保存后，作业票报警规则配置中可直接引用已定义的优先级级别。

作业票报警规则配置与启用。安全管理员进入作业票报警规则页面新增规则——从作业票类型列表中选择需要监控的作业票类别，填写规则编号和规则内容描述，然后为四种报警场景分别配置是否启用、阈值参数、优先级级别和分级通知人员链。规则保存后立即启用，系统开始对关联作业票类型的在执作业票进行实时报警判定。

报警中心统一汇聚与处置。控制室值班人员在报警中心通过左侧业务模块树筛选特殊作业报警类别——列表展示报警时间、报警类型、报警对象、报警内容、报警优先级（带颜色标签）、责任部门、确认/处理/消警状态和处理人信息。四个视图标签页区分待处理任务——值班人员在待办视图下逐条确认报警、填写处理说明后提交处理。

App 端报警接收与现场处置。现场管理人员在安卓 App 端接收报警推送通知——点击通知或从菜单进入报警列表，查看报警类型、时间、级别和简要内容。到达作业现场处置后在详情页点击处理按钮填写处置说明提交——处理结果实时同步至 Web 端报警中心，形成从报警生成到现场处置的完整闭环。

**对标要点**：
- 首段：一句话定位 + 业务范围（四种报警场景）+ 端形态（Web+App）+ 四类角色 + "其主要功能如下"自然引出
- 后续段落：按业务链路展开（标准建立→规则配置→报警汇聚→App 处置），每段是"谁→在哪→做什么→产出什么"的因果链
- 不用列表、不用编号、不用"、"。连接的功能名称
- 不用"旨在""赋能""高效便捷"等 AI 套话
- 每个系统特有的业务概念（X/Y/Z 矩阵、分级通知链）在简介中给出一句话定义

## 节选三：功能清单（第三章）

**样本原文**：

| 序号 | 功能模块 | 细分功能 | 功能说明 |
| --- | --- | --- | --- |
| 1 | 报警优先级配置 | 业务模块选择 | 从左侧"支持的业务模块"树中选择特殊作业报警对应的节点——右侧矩阵表格按所选模块刷新已定义的维度取值和优先级映射 |
| 2 | 报警优先级配置 | 维度配置 | 在弹出的维度配置对话框中分别设置纵向维度（Y 轴/响应时效）名称和取值列表、横向维度（X 轴/报警类别）名称和取值列表——取值列表支持拖拽排序和删除，至少保留一项 |
| 3 | 报警优先级配置 | Z 轴优先级配置 | 在弹出的报警级别配置对话框中添加优先级取值列表——每项配置颜色编码（颜色选择器）和名称（文本输入），支持拖拽排序 |
| 4 | 报警优先级配置 | 矩阵映射保存 | 在矩阵表格中为每个"Y 取值 × X 取值"交叉点从下拉框选择 Z 轴值——全部交叉点配置完成后点击保存按钮将 XYZ 映射写入数据库并以颜色编码展示 |
| 5 | 作业票报警规则 | 规则列表管理 | 进入作业票报警规则页面查看已创建规则——列表展示涉及票据类型、规则内容、启停状态和最近更新时间，支持新增、编辑和删除 |
| 6 | 作业票报警规则 | 选作业票类型 | 新增或编辑规则时在作业票类型选择区域从左侧备选列表勾选需纳入的作业票类别（含级别信息如动火-一级/动火-二级，仅显示特殊作业分类） |
| 7 | 作业票报警规则 | 四种报警场景配置 | 分别为作业人员离场（选择监控角色）、无关人员闯入、气体分析超时（设置检测间隔，仅动火和受限空间生效）、作业票超时配置启用开关和阈值参数 |
| 8 | 作业票报警规则 | 优先级与通知链 | 为每种启用的报警场景选择优先级级别，配置分级通知人员链（按人员/岗位/角色/部门多选推送对象，支持多级逐级升级），在消息配置对话框中设置推送模板内容 |
| 9 | 作业票报警规则 | 校验与启停 | 提交时校验涉及票据不能为空——规则保存后默认启用，可点击行内启停按钮控制规则生效状态——停用后不触发新报警但历史记录不受影响 |

**对标要点**：
- 每行 = 序号 + 功能模块（分组名）+ 细分功能（具体操作动作）+ 功能说明（30-80 字，一句讲清操作逻辑和业务含义）
- 细分功能粒度为"选作业票类型""四种报警场景配置""校验与启停"这种操作级别，不是"报警规则管理"这种大模块
- 功能说明要写出条件分支（"仅动火和受限空间生效"）、操作动作（"从左侧备选列表勾选"）、结果反馈（"以颜色编码展示"）
- 同一功能模块下按实际业务操作顺序排列（先选择模块→再配置维度→再设 Z 轴→最后映射保存）

## 节选四：台账型模块

> 台账型 = 标准 CRUD 列表页面。核心是字段完整性 + 业务规则精确性 + 异常覆盖全面性。

### 4A. 基础台账型

> **模块类型：台账型**（标准 CRUD，定位卡信息是实时地图可视化和出入记录管理的前置数据——只有已绑定人员的卡才能在地图上展示位置标记，只有未绑定的卡才能分配给来访人员）

操作路径：人员定位 → 定位卡管理。

本模块维护的定位卡信息是位置数据采集的前置条件——安全管理员在此批量导入或逐个录入定位卡，将卡绑定到具体人员后激活位置采集，人员离岗或卡损坏时执行解绑和重新分配。卡的状态和绑定关系直接影响地图上的人员展示和出入登记时的卡片可选范围。

#### 列表界面

| 功能点 | 描述 |
|---|---|
| 列表展示字段 | 定位卡号、门禁卡号、卡片电量、在线状态（在线/离线）、绑定状态（已绑定/未绑定）、绑定人员、人员类型、部门/单位、操作（绑卡记录、绑卡、解绑、删除） |
| 数据排序 | 默认按创建时间降序排序 |
| 默认每页数据量 | 10 条/页 |
| 查询功能 | 支持按定位卡号模糊查询、按绑定人员选择查询、按在线状态下拉筛选、按绑定状态下拉筛选、按部门下拉筛选、按承包商公司下拉筛选——点击"搜索"执行查询，点击"重置"清空条件 |
| 新增 | 点击"新增定位卡"按钮，弹出新增定位卡界面 |
| 绑卡记录 | 在操作列点击"绑卡记录"按钮——查看该定位卡完整的历史绑定和解绑记录，包括绑定人员、绑定时间和操作人 |
| 绑卡 | 对未绑定的卡，在操作列点击"绑卡"按钮——弹出人员选择界面搜索并选择目标人员后确认，卡状态切换为已绑定 |
| 解绑 | 对已绑定的卡，在操作列点击"解绑"按钮——确认后解除绑定关系，卡状态恢复为未绑定 |
| 删除 | 在操作列点击"删除"按钮——已绑定的卡需先解绑才能删除。支持勾选多条记录后批量删除。顶部工具栏同时提供"批量解绑"和"批量删除"按钮 |
| 批量操作 | 顶部工具栏提供"批量解绑"按钮——勾选多条已绑定记录后统一解绑。点击"更多"下拉菜单提供"下载模板""导入数据""导出数据"三项操作——导入操作按模板填写后上传，系统输出成功和失败行数统计 |

#### 新增/修改界面

| 字段名称 | 字段类型 | 是否必填 | 输入边界 | 业务规则 |
|---|---|---|---|---|
| 定位卡号 | 文本输入 | 是 | 限制数字、字母、横杠、冒号、下划线 | 唯一标识，不可与已存在的卡号重复 |
| 门禁卡号 | 文本输入 | 否 | 限制数字、字母、横杠、冒号、下划线 | 关联的门禁系统卡标识 |

#### 异常功能逻辑

| 异常情况 | 处理逻辑 |
|---|---|
| 新增定位卡时卡号重复 | 提示"卡号已存在，请重新输入"，阻止保存——光标回到卡号输入框 |
| 删除定位卡时该卡处于绑定状态 | 提示"该定位卡当前已绑定人员，请先解绑后再删除"，阻止删除 |
| 绑卡操作时无可选人员 | 提示"当前无可用人员数据，请先在人员管理中维护人员档案" |
| 必填字段为空 | 点击保存时校验——对应字段边框变红并提示"定位卡号不能为空" |
| 导入文件格式不符 | 提示"仅支持 Excel 格式的导入文件"，阻止上传 |
| 操作过程中网络中断 | 页面提示"网络连接中断，请检查网络后重试"，表单中已填写内容保留不丢失 |

**对标要点**：
- 列表界面表至少覆盖 10 个功能点：列表展示字段（逐一列名）、数据排序、分页、查询（精确/模糊方式都要写清）、新增、修改、删除、批量操作、特有操作（绑卡/解绑）、导出导入
- 新增/修改字段表 5 列齐全——字段类型精确（文本输入/下拉选择/日期选择器/开关/文件上传），业务规则写清校验逻辑和提示文案
- 异常至少覆盖 6 种——数据重复、关联约束阻止删除、前置数据缺失、必填为空、格式校验失败、网络中断
- 异常处理不只是"提示错误"，还要写阻止保存 or 阻止删除 or 保留填写内容

### 4B. 含嵌套子表的台账型

> 当台账型模块有嵌套的子配置界面时（如设备→检查项→检查标准），需要独立成节，逐层说明。

#### 检查项配置——本模块的核心操作

在列表操作列点击"检查项"——进入该设备的检查项管理界面。每台设备可配置多条检查项，每条检查项包含检查内容描述和对应的检查标准。

| 检查项字段 | 字段类型 | 是否必填 | 业务规则 |
|---|---|---|---|
| 项目名称 | 文本输入 | 是 | 检查项描述——如"法兰连接处是否存在泄漏迹象"或"压力表读数" |
| 检查标准（可添加多条） | 标准名称 + 判断类型 + 判定参数 | 是（至少一条） | 每条标准定义一个判定规则 |

**检查标准支持以下判断类型**：

| 判断类型 | 说明 | 示例 |
|---|---|---|
| 合格/不合格（定性判断） | 巡检人员目视或经验判断——无量化参数 | "目视检查法兰连接处无泄漏痕迹则合格——有渗漏痕迹则不合格" |
| 数值范围判断 | 填写判定参数阈值——巡检人员输入实测值后系统自动判定 | 标准名称"压力读数"——参数设为 ≥0.8 且 ≤1.2——实测值在该范围内则合格 |
| 下拉选择判断 | 预设选项列表——巡检人员从下拉列表中择一判定 | 标准名称"锈蚀程度"——选项为"无锈蚀/轻微锈蚀/中度锈蚀/严重锈蚀" |

**对标要点**：
- 嵌套子配置界面要说明"从哪进入→看到什么→每层能配什么→配置后下游怎么用"
- 判断类型/选项值等枚举型字段用独立表格列出，每项给出真实业务示例
- 层级关联关系在末尾用引用块总结（如"设备下配置的检查项在巡检点创建时自动带入"）

## 节选五：业务型模块

> 业务型 = 审批流转 / 任务执行 / 状态机 / 多角色协作。核心是对象生命周期完整 + 状态流转清晰 + 每步的操作者/前置条件/系统响应/异常处理齐全。

### 5A. 审批流转型

操作路径：承包商管理 → 承包商公司。

本模块维护的公司信息和审批备案结果是项目关联、开工审查、人员登记的前置条件——只有已备案的公司才能被下游模块引用。

#### 发起审批

| 操作步骤 | 用户操作 | 系统响应 |
|---|---|---|
| 1 | 在承包商公司信息填写完整后，点击"发起审批" | 系统将当前全部信息生成一份提交快照——包括基本公司信息、企业资质证书、主要管理人员、人员资质证书、保险缴纳凭证、内部培训资料、工器具信息、特种设备合格证书、过往服务记录、安全管理体系资料共十个业务模块。每个模块逐字段比对当前正式版本与快照版本的差异，生成变更明细 |
| 2 | — | 系统创建一条备案审批记录（状态为"审批中"），启动审批流程——签批节点和签批人由信息配置中定义的签批流程决定 |
| 3 | — | 审批任务推送到对应签批人的待办列表中，公司备案状态从"待备案"变更为"备案中" |

#### 公司备案状态

| 备案状态 | 含义 | 对应公司类型 |
|---|---|---|
| 待备案 | 公司信息已登记但尚未提交审批，或审批被终止后退回修改 | 准承包商 |
| 备案中 | 审批流程正在进行，数据快照已提交，等待签批人办理 | 准承包商 |
| 已备案 | 全部签批节点通过，快照数据覆盖为正式档案 | 合格承包商 |

#### 审批通过后的处理

| 处理项 | 变化 |
|---|---|
| 公司备案状态 | 从"备案中"变更为"已备案" |
| 公司类型 | 从"准承包商"升级为"合格承包商" |
| 数据更新 | 快照数据覆盖正式数据，旧正式数据转为审批历史记录存档 |
| 后续影响 | 该公司正式进入合格承包商名录库，可在项目管理和入场申请中被引用 |

**对标要点**：
- 审批发起用"操作步骤→用户操作→系统响应"三列表，纯系统动作的步骤用户操作列写"—"
- 状态表覆盖所有真实状态——每行写清楚状态名、含义和对应的业务标记
- 审批结果用"处理项→变化"对比表——读者一眼看到审批通过后什么东西变了

### 5B. 状态机型

隐患治理是巡检闭环的收口——从巡检不合格项自动创建待处理隐患，经方案制定、指派、整改、验收四步完成治理闭环。

#### 隐患治理状态机

| 状态 | 含义 | 触发条件 | 下一状态 |
|---|---|---|---|
| 待处理 | 隐患记录已创建——尚未制定治理方案 | 巡检不合格项自动创建或手动新增 | 处理中（安全管理员制定方案并指派） |
| 处理中 | 治理方案已制定——责任人已指派——正在整改 | 安全管理员提交治理方案 | 待验收（责任人提交整改结果） |
| 待验收 | 整改已完成——等待安全管理员验收 | 责任人提交整改结果 | 已关闭（验收通过）/ 处理中（验收退回） |
| 已关闭 | 隐患治理完成——验收通过 | 安全管理员确认验收 | 终态——不可修改——保留供审计 |

> **逾期规则**：超过整改期限未提交验收——状态保持为处理中——记录红色标记"已逾期"——安全管理员需跟进。

#### 隐患治理执行——完整四步链路

| 处理阶段 | 操作角色 | 操作内容 | 系统响应 | 触发下一阶段的约束 |
|---|---|---|---|---|
| 1. 制定方案 | 安全管理员 | 在待处理列表点击隐患记录——查看详情——填写治理方案——选择责任人——设置整改期限——提交 | 状态更新为处理中——任务推送到责任人 | 治理方案不可为空——责任人必选——整改期限必须设置为未来日期 |
| 2. 执行整改 | 责任人 | 进入待处理隐患列表——筛选分配给自己的任务——按治理方案执行现场整改——完成后填写整改结果和整改说明——提交验收 | 状态更新为待验收——安全管理员收到验收通知 | 整改结果不可为空——可上传整改照片或附件佐证 |
| 3. 验收确认 | 安全管理员 | 查看整改结果——确认整改是否达到验收标准——点击通过或退回 | 通过→状态更新为已关闭。退回→状态回退为处理中——需填写退回原因 | 必须填写验收意见——不可为空通过 |
| 4. 退回重治 | 责任人 | 查看退回原因——按原因补充整改——完成后重新提交 | 状态再次更新为待验收——进入下一轮验收 | 可覆盖更新原治理方案——整改期限可申请延长 |

**对标要点**：
- 状态机表 5 列：状态名 + 含义（一句话讲清当前阶段） + 触发条件 + 下一状态（含分支）
- 多阶段操作链路用"处理阶段→操作角色→操作内容→系统响应→约束"五列表——读者按列追踪每一步谁做什么、系统怎么响应、卡在什么条件
- 退出/分支路径必须注明（"验收退回"回到处理中）
- 逾期规则用引用块单独标注

## 节选六：混合型模块

> 混合型 = 既含台账维护（列表/新增/编辑/删除），又含业务配置或状态流转。核心是台账面和业务面各自独立叙述，两者之间的因果关系明确。

**样本原文（混合型模块——报警优先级配置）**：

> **模块类型：混合型**（台账面为优先级矩阵列表查看和维度取值的增删改查；业务面为通过维度配置→Z 轴级别配置→矩阵映射三个步骤建立完整的优先级矩阵——矩阵保存后被作业票报警规则引用）

操作路径：报警管理 → 报警优先级配置。

报警优先级配置是特殊作业报警体系的基础标准层。安全管理员在配置任何作业票报警规则之前需先在此处建立优先级矩阵——定义报警的严重程度分级（X 轴/报警类别）、响应速度分级（Y 轴/响应时效）以及每个交叉组合的最终优先级（Z 轴/优先级取值）。

页面布局为左右两栏：左侧面板标题为"支持的业务模块"，展示业务模块树供用户点击选择要配置的模块；右侧面板上方提供「维度配置」和「保存」两个按钮，下方为矩阵表格——表格的行对应 Y 轴的各个取值（首列显示取值文本），表格的列按 X 轴的取值展开（每列列名为取值文本）。每个"Y 行 × X 列"交叉点为一个下拉选择框——用户从此处选择 Z 轴优先级值。矩阵保存后各单元格以对应颜色编码展示。

#### 维度配置操作

点击「维度配置」按钮弹出对话框，分为左右两个卡片区域——左侧为"纵向维度配置"，右侧为"横向维度配置"。每个卡片内包含维度名称输入框和取值选项列表。

纵向维度（Y 轴）和横向维度（X 轴）分别独立配置。以纵向维度为例——先填写维度名称（如"响应时效"），然后在选项列表中逐项添加取值文本（如"立即响应""30 分钟内响应""2 小时内响应"）。取值列表每项为一个文本输入框，左侧有拖拽手柄支持调整顺序，右侧有删除按钮支持移除。维度名称和取值列表中每个取值的文本均不可为空——为空时保存提示对应错误。每个维度的取值列表至少保留一项，删除到只剩一项时提示"至少保留一个选项"。

#### Z 轴优先级级别配置

点击页面顶部卡片内的「报警优先级配置」按钮弹出第二个对话框（标题为"报警级别配置"）。对话框内为 Z 轴优先级取值列表——每项包含一个颜色选择器和一个文本输入框，支持拖拽排序和删除。每个取值定义一个优先级级别——例如"一级紧急报警"并设置红色（#FF4D4F）、"二级重要报警"并设置橙色（#FAAD14）、"三级一般报警"并设置蓝色（#1677FF）。

#### 矩阵映射与保存

两项配置完成后回到主页面，矩阵表格的行按 Y 轴取值展开、列按 X 轴取值展开。用户在每个交叉点的下拉框中从 Z 轴取值列表中选择对应的优先级值。全部交叉点配置完成后点击「保存」按钮——系统将 XYZ 映射关系写入数据库，矩阵表格刷新展示已保存的优先级值并以颜色编码显示。

#### 异常功能逻辑

| 异常情况 | 处理逻辑 |
|---|---|
| 维度名称（纵向或横向）为空 | 点击确定时提示"请输入纵向维度名称"或"请输入横向维度名称"——阻止保存 |
| 选项取值内容为空 | 点击确定时提示"请填写纵向维度值"或"请填写横向维度值"——阻止保存 |
| 选项列表为空 | 点击确定时提示"请至少添加一个纵向/横向维度选项"——阻止保存 |
| Z 轴取值列表为空 | 点击确定时提示"请至少添加一个报警级别"——阻止保存 |
| 删除维度取值时该取值被作业票报警规则引用 | 弹出确认对话框——确认删除后系统同时解除规则中的引用关系 |
| 列表仅剩一项时执行删除 | 提示"至少保留一个选项"——阻止删除 |

**对标要点**：
- 混合型模块的模块类型标注要同时说明台账面是什么、业务面是什么、两者之间的因果关系
- 多步骤配置流程按先后顺序分节叙述——第一步打开什么对话框→第二步配置什么→第三步映射保存
- 每个对话框的 UI 布局要在操作说明前给出一句话总述（如"对话框分为左右两个卡片区域"）
- 条件显隐规则要写明（"气体分析超时报警仅对动火和受限空间生效"）
- 异常覆盖对话框级校验（维度名称为空、选项为空、列表为空、引用约束）

## 节选七：App 端模块

> App 端模块优先使用操作步骤表，说明界面切换、硬件交互和失败处理。不套用 Web 端的"字段表"模式。

**样本原文（App 端模块——报警处置）**：

> **模块类型：业务型**（App 端报警处置——现场管理人员接收推送通知后查看报警详情并到现场处置后在线反馈——处理结果通过实时通信同步至 Web 端形成完整处置闭环——报警列表仅展示推送至当前用户的报警——未被配置为通知对象的用户列表为空）

现场管理人员作为本模块的适用用户——通过安卓 App 接收作业票报警规则触发的报警推送通知。用户登录 App 后 App 保持与后端服务的实时连接——报警生成后系统按规则配置的通知链将报警信息推送至对应人员的手机。用户点击推送通知或从 App 菜单进入报警列表——列表展示当前用户相关的特殊作业报警记录，每项显示报警类型、报警时间、报警级别和简要内容，支持下拉刷新获取最新数据。

报警记录有三种状态——未读（新推送尚未查看）、已读（已点开查看详情但尚未处置）、已处理（已提交处置反馈）。

#### App 端处置操作步骤

| 操作步骤 | 用户操作 | 系统响应 | 异常处理 |
| --- | --- | --- | --- |
| 1 | 现场管理人员点击推送通知或从 App 菜单进入报警列表——下拉刷新获取最新数据 | 系统展示当前用户相关的特殊作业报警——每项显示报警类型、时间、级别和简要内容 | 若列表为空——当前用户未被任何作业票报警规则配置为通知对象——联系管理员确认通知链配置 |
| 2 | 点击某条报警进入详情页——查看报警完整信息和作业区域位置 | 系统将报警标记为已读——状态同步至后端——用户获得判断是否需要现场处置的充分信息 | 若作业区域位置为空——作业票创建时未填写作业地点——到作业票管理模块补充后新报警使用更新地点 |
| 3 | 到达作业现场处置完成后在详情页底部点击「处理」按钮——填写处置说明后提交 | 系统将报警状态变更为已处理——结果通过实时通信同步至 Web 端报警中心——值班人员即时可见处理结果 | 若提交时提示无操作权限——该报警不在当前用户处置权限范围内——确认当前用户是否在该报警的通知处理链中 |

**对标要点**：
- App 模块用"操作步骤→用户操作→系统响应→异常处理"四列表，不套用字段表
- 每步的用户操作写清具体动作（点击通知→进入详情→填写说明→提交），系统响应写清状态变化和同步机制
- 异常处理列要写清"用户应该做什么"——不只是"提示错误"，还要给出排查路径
- 说明段落要点明端特色——推送通知、实时同步、下拉刷新、离线暂存
- 与 Web 端的闭环关系在正文中写清（"处理结果实时同步至 Web 端报警中心"）

## 节选八：大屏模块

> 大屏模块说明页面布局、数据来源、刷新方式和用户可见结果，不套用新增修改字段表。

**样本原文（大屏模块——实时态势展示）**：

> **模块类型：业务型**（实时数据大屏，通过 WebSocket 长连接推送数据——展示人员热力图、告警滚动列表、区域人数统计和告警趋势折线图四个数据区域，无需手动刷新）

操作路径：从系统菜单进入大屏展示页面。

本模块是面向企业负责人的全局态势感知页面——通过 WebSocket 长连接将位置数据、告警事件和人数统计实时推送到大屏四个数据区域。企业负责人在指挥中心打开大屏页面即可了解全厂人员分布热力图、告警动态滚动、各区域实时人数统计和当日告警趋势分析。

#### 大屏四个数据区域

| 区域 | 位置 | 展示内容 | 刷新方式 |
|---|---|---|---|
| 人员热力图 | 左侧 | 3D 厂区地图叠加实时热力图——红色表示人员密集、蓝色表示稀疏——鼠标悬停可查看该区域具体人数 | 每 2 秒自动刷新 |
| 告警滚动列表 | 底部 | 滚动展示当日所有告警事件——新告警触发时自动滚动到最新条目并以高亮闪烁提示——不同告警类型用不同颜色标签区分 | 实时推送 |
| 区域人数统计 | 右侧 | 各区域实时人数统计卡片——展示区域名称、当前人数、配置的人数上限和下限——超限时卡片边框变红并持续闪烁 | 每 2 秒自动刷新 |
| 告警趋势折线图 | 底部 | 按小时统计当日各类告警数量——不同告警类型用不同颜色绘制——鼠标悬停可查看具体时段数据 | 每 5 分钟自动刷新 |

#### 异常功能逻辑

| 异常情况 | 处理逻辑 |
|---|---|
| WebSocket 连接断开 | 页面中央显示"数据连接已断开，正在尝试重新连接……"——连接恢复后数据自动刷新。连续重试 5 次失败后提示"请检查网络连接或联系系统管理员" |
| 地图模型文件加载失败 | 热力图区域降级展示 2D 平面图——热力图叠加功能不受影响。页面右上角短暂提示"3D 模型未加载，已切换至 2D 视图" |
| 浏览器不支持 WebGL 渲染 | 页面加载时检测到不支持——中央提示"当前浏览器不支持 3D 渲染，请使用 Chrome 90+ 或 Edge 90+ 浏览器" |
| 无当日告警数据 | 告警滚动列表显示"今日暂无告警事件"——告警趋势折线图显示 Y=0 的平直线 |

**对标要点**：
- 大屏模块用"布局区域表"（区域/位置/展示内容/刷新方式）替代字段表——只说明用户看到什么、数据怎么更新
- 每个数据区域的视觉状态写清（红色=密集、蓝色闪烁=超限、高亮闪烁=新告警）
- 异常覆盖 Websocket 断开、模型加载失败、浏览器兼容性、无数据四种典型大屏场景
- 不写数据接口、不写 WebSocket 握手协议、不写渲染引擎

## 节选九：典型使用流程（第七章）

> 典型使用流程串联跨模块、跨角色、跨端链路。选一个完整的业务场景从头走到尾——每步写清谁在哪个页面做什么、系统怎么响应、产出什么供下一步使用。

**样本原文——一次动火作业从规则配置到报警处置的完整过程**：

**场景：某化工企业计划在厂区 A 号车间进行为期 3 天的动火作业——涉及高处和地面两个工作面。开工前一天安全管理员完成报警规则配置，作业开始后系统自动监测异常。**

第一步——安全管理员登录 Web 端进入报警优先级配置页面。点击「维度配置」按钮——在弹出对话框中分别设置纵向维度（Y 轴/响应时效）取值"立即响应""30 分钟内响应""2 小时内响应"，横向维度（X 轴/报警类别）取值"一级紧急报警""二级重要报警""三级一般报警"。再点击「报警优先级配置」按钮——在弹出对话框中添加三个 Z 轴优先级级别——"高级"（红色 #FF4D4F）、"中级"（橙色 #FAAD14）、"低级"（蓝色 #1677FF）。回到主页面在矩阵表格中为每个交叉点选择 Z 轴值——"一级紧急 × 立即响应"选高级。全部配置完成后点击「保存」按钮——矩阵以颜色编码展示已保存的优先级映射。

第二步——安全管理员进入作业票报警规则页面点击「新增」。在作业票类型选择区域从左侧备选列表中勾选"动火-一级""动火-二级"。在四种报警场景配置区中依次配置——作业人员离场报警：启用并选择监控角色为作业监护人和作业人员；气体分析超时报警：启用并设置动火气体检测间隔为 120 分钟（2 小时）；作业票超时报警：启用并设置提前 30 分钟预警。为每种启用场景分别选择 X 轴"一级紧急报警"和 Y 轴"立即响应"，系统自动回填 Z 轴"高级"和颜色。在通知链配置区中添加三个分组——第一级通知对象选择"作业监护人角色"、第二级选择"车间安全负责人角色"、第三级选择"安全管理部角色"。提交保存后规则立即启用。

第三步——动火作业开始执行。后端监控引擎每 5 秒轮询一次——通过人员定位数据判断作业监护人和作业人员是否在作业区域范围内、通过气体检测记录表判断最近一次检测距当前的时间是否超过 120 分钟。第二天下午——距离上次气体检测已过 2 小时 15 分钟仍未上传新数据——系统判定满足气体分析超时报警条件。自动生成一条报警记录——报警类型为"气体分析超时报警"——报警对象为"A 号车间动火作业"——报警优先级为"高级"（红色）。按规则的通知链向三级人员推送——第一级"作业监护人"角色内的所有在线人员收到 Web 端和 App 端推送。

第四步——控制室值班人员在报警中心的"我的待办"视图下看到一条新的待确认报警记录。值班人员勾选该记录点击「批量确认」——确认状态按钮变为绿色"已确认"——确认人员列显示当前值班人员姓名和确认时间。

第五步——作业监护人在安卓 App 上收到推送通知。点击通知进入报警详情页——查看报警时间、类型为"气体分析超时报警"、作业票名称为"A 号车间动火作业"、作业区域位置为"A 号车间焊接工位"和超时时长 15 分钟。监护人立即安排气体检测人员携带便携式可燃气体检测仪到 A 号车间焊接工位进行气体复检。检测合格后——监护人在详情页底部点击「处理」按钮——填写处置说明"可燃气体复检结果为 0.02%LEL——远低于安全阈值——已确认作业区域通风正常——可继续作业"。提交后系统将报警状态变更为已处理——结果实时同步到 Web 端报警中心。

第六步——Web 端报警中心中该报警的处理状态变为绿色"已处理"。新的气体检测数据上传成功后——系统检测到最近一次检测时间距当前已小于 120 分钟——报警条件解除——自动执行消警——消警状态变为绿色"已消警"——记录自动消警时间。

第七步——作业第三天傍晚作业票提交完成确认。后端检测到作业票状态变更为已完成——该票对应的作业票超时报警规则自动停止对其的监测。整个特殊作业报警管理生命周期闭环完成——从规则配置到报警触发、确认分发、现场处置、自动消警全过程在系统内完整可追溯。

**对标要点**：
- 在流程开头用一句加粗的场景描述设定上下文（什么作业、在哪、多长时间、涉及谁）
- 每步以"第 X 步——角色 + 进入什么页面 + 做什么"开头，系统动作以"—"连接
- 时序贯穿——第一步优先级配置→第二步规则创建→第三步报警触发→第四步值班确认→第五步 App 处置→第六步消警→第七步结束
- 跨角色切换在每步开头标注（安全管理员→值班人员→作业监护人）
- 具体数据写活场景——"120 分钟""2 小时 15 分钟""0.02%LEL""A 号车间焊接工位"
- 每一步的操作结果自然衔接到下一步的操作前提

## 节选十：术语表

**样本原文**：

| 术语 | 含义 |
| --- | --- |
| 特殊作业 | 危险化学品企业中动火、受限空间、高处、吊装等需要办理作业票的高风险作业，执行标准 GB 30871 |
| 作业票 | 特殊作业的安全许可凭证——包含作业类型、作业区域、作业人员、监护人、气体检测要求、计划时间和有效期等信息 |
| 报警规则 | 安全管理员配置的报警判定标准——含适用作业票类型、四种报警场景的启用状态、阈值参数、优先级和分级通知人员链，规则启用后系统立即开始监测 |
| X/Y/Z 优先级矩阵 | X 轴为报警类别维度（严重性指标），Y 轴为响应时效维度（响应速度指标），Z 轴为每个 X/Y 交叉点对应的最终优先级取值——以颜色编码区分视觉等级 |
| 分级通知 | 按优先级配置的多级通知机制——每级独立指定推送对象（人员/岗位/角色/部门），上一级在设定时限内无人响应时系统自动升级到下一级推送 |
| 作业人员离场报警 | 作业进行期间监护人、负责人或作业人员在未完成作业的情况下离开作业区域的报警——可在规则中选择监控角色 |
| 气体分析超时报警 | 动火和受限空间作业中距上次气体检测时间超过规定间隔的报警——仅对这两种作业类型生效 |
| 消警 | 当触发报警的异常条件恢复正常后系统自动取消报警的操作——记录消警时间但不删除报警记录——已消警记录仍可在全部报警视图中查询和导出 |

**对标要点**：
- 每个术语一句讲清"是什么、在系统中怎么用"——不写词典式通用定义
- 术语要是系统特有的业务概念（不是通用的"用户名""密码""登录"），每个功能模块至少贡献一个术语
- 如果术语有特定适用条件，必须在含义中标注（如"仅对动火和受限空间生效"）
- 缩写/编码型术语（如 X/Y/Z）先展开全称再给定义
- 放在系统要求之后、功能操作之前（第 5 章）

## 节选十一：FAQ

> FAQ 不是教科书式定义，而是**用户遇到具体问题 → 为什么会出现 → 怎么解决**。

### Web 管理端 FAQ

**问：配置了作业票报警规则并启用后——报警中心列表中始终没看到报警记录——怎么排查？**

答：按以下顺序逐项排查。第一步——进入作业票报警规则列表检查规则状态是否为启用——行内「启停状态」标签显示"停用"则表示规则未生效。第二步——确认该规则关联的作业票类型下是否存在正在执行中的作业票——只有正在执行（非草稿、已完成或作废）的作业票才会被监测。第三步——如果规则已启用且作业票在执行中但超时报警仍未触发——检查当前时间是否已超过配置的阈值参数。第四步——检查通知链配置中是否指定了推送对象——如果推送对象选的是某角色但该角色下无人员——通知实际不会到达任何人——但报警记录已在报警中心生成——切换到"全部报警"视图确认是否存在。

**问：四种报警场景可以只启用一部分吗——不启用的场景会影响其他场景吗？**

答：四种场景独立开关——每种场景的配置和判定完全独立。不启用的场景完全不参与报警判定——不消耗系统资源。

### App 端 FAQ

**问：为什么 App 收不到报警推送通知？**

答：依次检查三项。一——App 登录的账号是否在 Web 端作业票报警规则的通知链中。通知链配置中的推送对象是"角色"时——当前 App 登录的人员必须属于该角色——否则不会收到推送。二——手机系统设置中 App 的通知权限是否已开启。三——App 是否为最新版本。以上均正常仍无法收到推送时联系管理员检查后端推送服务运行状态。

**问：App 端处理报警后 Web 端值班人员能看到吗？**

答：能——App 提交处理结果后数据通过实时通信同步至后端——Web 端报警中心列表和待处理看板的统计数字即时刷新。处理结果中填写的处置说明也会展示在 Web 端报警详情弹窗的"处理/忽略说明"字段中。

### 跨岗位 FAQ

**问：安全管理员和值班人员的菜单权限有什么区别？**

答：安全管理员可访问报警优先级配置和作业票报警规则——可新增、修改、删除规则和矩阵。值班人员可访问报警中心和待处理报警看板——可确认、处理、忽略和导出报警记录——但不能修改规则配置。企业负责人通常只开放待处理报警看板的查看权限——不参与具体报警处置。

**对标要点**：
- 至少 9 问，分 Web 端 / App 端 / 跨岗位三层
- 每问从用户实际困惑出发——"收不到""看不到""怎么办""有什么区别"
- Web 端 FAQ 重在排查路径——给出按顺序逐项检查的步骤
- App 端 FAQ 重在端间协同——"App 处理了 Web 端能看到吗"
- 跨岗位 FAQ 重在权限和职责边界——"谁能做什么"
- 回答中引用具体操作路径和页面名称——用户能按图索骥找到对应按钮

## 节选十二：跨端系统章节组织

> 当系统包含 Web 管理端、大屏展示端、Android App 端等多个客户端时，功能操作章节按端拆分为独立的章。

**正例**（Web + 大屏 + App 三端系统）：

```
6 Web 管理端功能操作
  ├─ 6.1 登录界面
  ├─ 6.2 定位卡管理
  ├─ 6.3 实时位置地图可视化
  ├─ 6.4 厂区地图配置
  ├─ 6.5 摄像头管理
  ├─ 6.6 外部人员出入记录管理
  └─ 6.7 位置告警管理
7 大屏展示端操作
  └─ 7.1 大屏实时态势展示
8 Android App 端操作
  └─ 8.1 App 端定位卡识别与位置校验
9 典型使用流程
10 常见问题解答
```

**反例**（将多端内容堆在一个章里）：

Web 管理端功能操作下包含所有 11 个模块——大屏热力图和 App NFC 识别的操作描述混杂在 Web 端的表格和字段列表中。审核员无法快速找到"大屏上怎么用""手机上怎么操作"。

## 对照清单：操作手册是否达标

**软著强制章节**：
- [ ] 登录界面：Web 管理端功能操作章首节——含截图占位 + 账号/密码/登录操作说明
- [ ] 系统首页（如存在）：登录界面之后——含截图占位 + 布局描述

**叙事密度**：
- [ ] 系统简介按业务链路自然段叙事，每段有因果链（谁→做什么→产出什么→下游谁接着用）
- [ ] 正文中每个模块提到了适用用户角色
- [ ] 不用"旨在/赋能/一站式/高效便捷/显著提升/强大能力"等 AI 套话

**表格深度**：
- [ ] 新增/修改界面表 5 列齐全（字段名称/字段类型/是否必填/输入边界/业务规则）
- [ ] 业务规则写了校验逻辑和具体提示文案（不是"校验不通过提示错误"这种空话）
- [ ] 条件显隐的字段在业务规则中标注触发条件

**模块类型特征**：
- [ ] 业务型模块有状态流转表或生命周期说明
- [ ] 混合型模块台账面和业务面各自独立叙述，因果关系明确
- [ ] App 端模块用操作步骤表（四列），不套用字段表
- [ ] 大屏模块用布局区域表，不套用新增修改字段表

**异常覆盖**：
- [ ] 每个模块异常至少覆盖 5 种——数据重复、关联约束、必填为空、格式校验、网络中断
- [ ] 异常处理不只是"提示错误"，要写阻止/允许/保留数据/跳转

**典型使用流程**：
- [ ] 至少一个完整场景的跨角色、跨模块、跨端操作链路
- [ ] 场景有具体业务上下文（时间、地点、人物、事件）
- [ ] 每一步的操作结果衔接到下一步的操作前提

**全局**：
- [ ] 总字数 ≥ 20,000
- [ ] 多端系统：功能操作章节按端拆分独立成章
- [ ] 零技术词（Vue/Kotlin/Spring Boot/MyBatis/API/JSON/SQL）——系统要求章节除外
- [ ] FAQ ≥ 9 问，分 Web/App/跨角色三层
- [ ] 术语表覆盖所有核心业务概念，不放通用词
