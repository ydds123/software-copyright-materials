# 操作手册构建规范

本文件是操作手册正文构建规则的唯一真相来源，只回答“怎么写”。审查方法与 Gate 映射见 `manual_quality_spec.md`。

## 1. 总体目标

操作手册应像真实软件随附的用户说明书。每段内容都应回答至少一个具体问题：谁在什么场景进入哪个页面、看到什么、执行什么、受到什么限制、完成后得到什么反馈或后续流转。

正文必须基于已确认的 `草稿/业务理解.json` 和真实项目证据。不得根据通用产品经验补造当前项目不存在的功能。

## 2. 全文章节结构

使用阿拉伯数字层级编号：

1. 系统简介
2. 系统概述
3. 功能清单
4. 系统要求
5. 术语表
6. 按客户端组织的功能操作章节
7. 典型使用流程
8. 常见问题解答

多端系统应将 Web、App、大屏等客户端分别设为一级功能操作章。登录界面和真实存在的系统首页放在 Web 管理端功能操作章开头。

各章职责必须互斥：

- 系统简介：连续叙述软件定位、角色和完整业务链路。
- 系统概述：说明目标、范围、产品组成和整体结构。
- 功能清单：汇总模块、细分功能和业务作用，不展开操作步骤。
- 系统要求：说明用户使用和部署所需环境。
- 术语表：统一解释核心实体、状态、角色和操作名称。
- 功能操作：按真实页面或核心流程说明用户操作。
- 典型使用流程：串联跨模块、跨角色、跨端链路。
- FAQ：回答真实使用中的问题，不重复正文。

每个 `##`、`###`、`####` 标题前至少保留一个空行。

## 3. 全文写作约束

- 使用面向普通用户的客观中文，不写代码实现、函数名、接口路径、框架、数据库字段或内部标识符。
- 使用术语标准表中的唯一名称，不混用同义词。
- 每个章节必须包含段落化说明，不能只有标题、项目符号或表格。
- 功能特点和系统简介按业务链路展开，不采用“总述 + 编号列表 + 总结”的统一模板。
- 不使用“旨在、赋能、一站式、智能化、高效便捷、显著提升、强大能力、丰富功能”等空泛表达。
- 不使用“进入方式、页面内容、操作规则、操作结果与反馈”等制式字段标题堆叠正文。
- 测试代码、注释代码和旧项目样本文案不能作为正常业务行为证据。

## 4. 功能清单

功能清单至少包含：序号、功能模块、细分功能、功能说明。

- 每个模块的细分功能必须来自真实页面、按钮、流程或业务能力。
- 功能说明应写清角色、业务目的、关键动作和结果，不能只列 CRUD 动词。
- 完成模块正文后必须回头升级功能清单，将状态流转、条件分支和上下游影响补入业务型模块说明。
- 功能清单中的每个子功能必须在后续操作章节中有对应描述。

## 5. 功能模块通用结构

每个模块至少包含：

1. 模块名称和模块类型。
2. 真实操作路径。
3. 使用场景、适用角色、上游前提和下游影响。
4. 页面用途及用户可见内容。
5. 用户动作、输入限制、系统反馈和异常处理。
6. 截图或清晰截图预留。

操作路径优先来自菜单 SQL 的 `sys_menu` 层级，其次来自前端路由 `meta.title`。无法取得时统一使用“从系统菜单进入对应功能页面”。

## 6. 按模块类型决定深度

### 6.1 台账型

适用于列表查询、新增、修改、删除、导入、导出等数据维护页面。

必须说明：

- 列表中的真实筛选项、展示列、顶部操作和行操作，不使用“等”字省略。
- 新增和修改时的字段、类型、必填条件、输入边界和业务规则。
- 至少说明保存失败、重复数据、关联占用、权限不足等真实异常。
- 说明该台账被哪些下游模块引用，以及错误数据会产生什么影响。

### 6.2 业务型

适用于审批、任务执行、状态流转、多角色协作和跨端闭环。

必须说明：

- 对象从创建到结束的生命周期和状态流转。
- 每个阶段的操作者、前置条件、用户动作、系统响应和异常处理。
- 每条条件分支、退回路径、终止路径和后续影响。
- 状态机、条件矩阵、处理项变化或跨角色交接应使用适合的表格表达。

### 6.3 混合型

同时包含台账维护和业务配置或状态流转时：

- 台账部分按台账型规则编写。
- 业务部分按业务型规则编写。
- 明确配置数据怎样影响后续任务、审批或执行流程。

### 6.4 App 和大屏

- App 模块优先使用操作步骤表，说明界面切换、拍照、NFC、GPS 等真实交互和失败处理。
- 大屏模块说明页面布局、数据来源、刷新方式和用户可见结果，不套用新增修改字段表。

## 7. 表格规则

- 表格用于帮助用户比较、判断、执行或确认结果，不为凑版面而创建。
- 同一表格同一列的内容必须具备行级信息增量；重复内容应按当前行上下文重写或删除无价值列。
- 操作步骤表使用“操作步骤、用户操作、系统响应、异常处理”四列。
- 字段表只保留有信息增量的列；整列均为“无”时删除该列。
- 技术术语必须翻译成用户能理解的业务语言。

## 8. 图片与截图位置

图片必须与它说明的内容紧邻：

- 操作流程图：放在模块功能说明之后、第一个细节子节之前。
- 列表截图：放在列表界面描述附近。
- 新增或修改截图：放在新增修改字段表之后、异常逻辑或下一操作阶段之前。
- 登录截图：放在登录说明文字之后。
- 详情或审批截图：放在对应描述之后。

禁止将截图统一堆放在模块末尾。用户跳过截图时，保留 `【截图预留：请在此处插入 xxx 页面或操作结果截图。】`。

## 9. FAQ 与术语表

- FAQ 至少覆盖 8 个真实问题，并覆盖不同客户端和跨岗位协作。
- FAQ 回答必须与正文中的操作步骤、状态和异常处理一致。
- 术语表解释当前项目中的核心实体、状态、角色和操作，不收录无关技术词。

## 10. 禁止事项

- 编造菜单路径、字段、按钮、状态、角色或流程。
- 把源码标识符、API、SQL、数据字典键名直接写入正文。
- 将不同模块统一套入相同句式。
- 用范本或旧项目内容替代当前项目证据。
- 用流程图替代必要的文字说明和操作步骤。
